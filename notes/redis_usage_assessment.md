# Redis 使用现状与合理性评估

适用场景：

- 需要快速了解当前项目哪些地方使用了 Redis
- 需要判断当前 Redis 设计是否合理
- 需要为后续是否重构 Redis 使用方式提供依据

本文基于当前仓库代码整理，不讨论假设性未来方案，只评估当前实现。

## 1. 当前 Redis 使用范围总览

当前项目里，Redis 主要用于以下 3 类场景：

1. 登录会话中心
2. 导入预检结果缓存
3. 分布式锁

其中：

- 登录会话中心已经完成新方案改造
- 导入预检缓存当前保持现状
- 分布式锁当前继续保留

## 2. 本次评估范围

本次重点评估的是登录之外的另外两类 Redis 使用方式：

1. 导入预检缓存
2. 分布式锁

结论先给出：

- 导入预检缓存：当前阶段合理，继续保持现状
- 分布式锁：设计合理，建议继续保留

## 3. 导入预检缓存评估

### 3.1 代码位置

- [import.service.ts](/D:/Sinhe/api/apps/api/src/import/import.service.ts)

关键使用点：

- 预检写入：`setJson(this.getPreviewKey(previewId), snapshot, IMPORT_PREVIEW_TTL_SECONDS)`
- 预检读取：`getJson<PreviewSnapshot>(this.getPreviewKey(previewId))`
- key：`import:preview:{previewId}`
- TTL：`3600` 秒，即 1 小时

### 3.2 当前用途

当前导入链路里，Redis 在这里承担的是“短期临时缓存”职责：

1. 前端先发起导入预检
2. 后端计算预检结果
3. 将预检快照写入 Redis
4. 前端拿到 `previewId`
5. 正式提交时，后端再从 Redis 读取该快照继续执行

快照内容包括：

- 模板命中结果
- 聚合后的订单草稿
- 重复订单信息
- 无效行
- 汇总统计信息

### 3.3 为什么说当前阶段合理

这套方案当前是合理的，原因如下：

1. 预检结果本身是临时数据，不属于长期事实源
2. 放 Redis 比单独落数据库表更轻，联调和迭代效率更高
3. 能避免“预检后提交”之间重复执行预检计算
4. 当前项目阶段仍处于稳态迭代和联调补强，Redis 缓存方案足够支撑现阶段需求
5. 代码里已经校验 `snapshot.tenantId !== tenantId`，避免跨租户直接复用预检结果

### 3.4 当前方案的局限

虽然当前合理，但它不是长期最优设计，主要局限包括：

1. Redis 不是审计存储，快照过期或被清空后无法回放
2. 如果预检快照很大，会直接占 Redis 内存
3. 目前校验维度是租户级，而不是“预检创建用户”级
4. Redis 重启、TTL 到期或手工清理后，用户可能出现“预检刚做完但提交时找不到结果”的情况

### 3.5 当前结论

当前结论是：

- 这块设计在当前阶段是合理的
- 暂时不改
- 继续保持 Redis 预检缓存方案

只有在以下场景出现后，才值得考虑升级：

1. 导入量明显增大
2. 需要审计留痕和历史回放
3. 需要更强的失败恢复能力
4. 需要严格做到“谁预检，谁提交”

## 4. 分布式锁评估

### 4.1 代码位置

- [redis.service.ts](/D:/Sinhe/api/apps/api/src/redis/redis.service.ts)
- [payment.service.ts](/D:/Sinhe/api/apps/api/src/payment/payment.service.ts)
- [import.service.ts](/D:/Sinhe/api/apps/api/src/import/import.service.ts)

### 4.2 当前使用点

当前 Redis 锁主要用于 3 条链路：

1. 在线支付发起锁
2. 线下支付登记锁
3. 导入任务消费锁

对应 key：

- `payment:initiate:{orderId}`
- `payment:offline:{orderId}`
- `import:job:lock:{jobId}`

### 4.3 当前用途

Redis 在这里承担的是“并发互斥控制”职责。

#### 在线支付发起锁

目的：

- 防止同一订单被重复发起在线支付
- 避免短时间内重复创建支付单

#### 线下支付登记锁

目的：

- 防止同一订单被重复登记线下支付
- 避免现金登记和其他已付登记并发写入

#### 导入任务消费锁

目的：

- 防止同一个导入任务被多个 Worker 或轮询执行器重复消费
- 避免任务重复落库

### 4.4 为什么说设计合理

当前锁设计总体合理，原因如下：

1. 支付本身就是高并发风险点，不加锁容易重复写入支付状态
2. 导入任务是异步消费模型，不加锁容易重复处理同一任务
3. 锁值不是裸删，而是“获取唯一 lockValue -> Lua 校验后释放”，这比简单 `DEL` 更安全
4. Redis 锁不是唯一保护措施，后面还有数据库事务和状态判断作为第二层保护
5. 当前支付链路执行时间通常较短，锁 TTL 能覆盖主要处理窗口

可以把当前方案理解为：

- Redis 锁负责第一道并发闸门
- 数据库事务和状态机负责第二道业务正确性闸门

这个组合是合理的。

### 4.5 当前方案的注意点

当前锁方案有一个工程化上的可增强点：

1. 目前锁是固定 TTL，没有续租机制
2. 如果未来某些任务执行时间显著超过 TTL，锁可能提前失效

不过从当前业务看：

- 支付链路通常足够短，风险可控
- 导入任务虽然更重，但任务本身还有 `heartbeatAt`、状态机和数据库控制，不是只靠 Redis 锁单独兜底

因此这不是当前必须立刻改的问题。

### 4.6 当前结论

当前结论是：

- 分布式锁设计合理
- 当前应该继续保留
- 后续如果导入处理时间显著增长，再评估锁续租机制

## 5. 最终结论

基于当前项目阶段与当前代码实现，本次评估给出的明确结论如下：

1. 登录会话中心已按新的会话模型改造完成
2. 导入预检缓存当前保持现状，不改
3. 分布式锁当前继续保留

## 6. 建议的后续口径

后续如果再讨论 Redis 改造，建议遵守以下口径：

1. 登录会话中心按当前会话模型继续迭代
2. 导入预检缓存除非出现明确业务压力，否则不主动重构
3. 分布式锁作为支付与导入链路的并发闸门继续保留
4. 若未来要做 Redis 专项治理，优先评估导入预检缓存，而不是先动分布式锁
