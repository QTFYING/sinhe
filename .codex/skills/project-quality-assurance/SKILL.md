# 项目质量闸门

在进入下一批开发前做一次集中复核，避免把局部正确、全局打架的实现继续往下推进。

## 入口约束

1. 先遵守仓库根目录 `AGENTS.md`。
2. 复核范围以 `design` 执行手册中的当前批次为准，不把暂缓域强行并入本轮验收。
3. 发现事实源冲突时，按 `docs/api -> enums -> contracts -> data-model-reference -> code -> Swagger` 的顺序回溯。

## 必查清单

1. 多租户隔离
   所有租户侧接口是否都按 token 中的 `tenantId` 隔离。
2. 金额正确性
   订单金额、折扣、实收、支付流水是否自洽。
3. 状态机
   文档里的状态值、实现里的状态判断、数据层保存值是否一致。
4. 幂等
   仅真正高风险链路实现幂等，且幂等键有物理或逻辑保障。
5. 文档一致性
   `docs/api`、`enums`、`contracts`、`data-model-reference` 是否仍是同一套语义。
6. 历史残留
   是否还存在旧字段名、旧接口路径、旧状态值或已放弃的旧链路。

## 推荐执行顺序

1. 先扫文档和类型定义。
2. 再看 Prisma schema。
3. 再看 service 状态流转。
4. 最后看 controller 输出是否与 contracts 对齐。

## 高风险信号

- 旧字段名与新字段名同时存在。
- 订单状态与支付状态互相覆盖。
- 打印接口重新演化回 `/print/jobs` 语义。
- H5 支付链路和 Tenant 核销链路使用不同金额口径。
- 历史归档文件重新被当成事实源。

## 完成标准

- 当前批次改动能说明“为何正确”而不只是“能编译”。
- 主要业务红线没有明显漏洞。
- 可以作为下一批开发的稳定基线。
