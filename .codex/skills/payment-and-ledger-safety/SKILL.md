# 支付账务安全

金额正确性优先于接口速度。支付相关逻辑必须先证明状态流转和入账规则成立，再考虑实现细节。

## 入口约束

1. 先遵守仓库根目录 `AGENTS.md`。
2. 支付域实施前，先确认 `design` 手册是否已经进入订单域和 H5 支付域阶段。
3. 不沿用旧 `payStatus` 主流程，不让历史 MVP 支付实现反向定义当前链路。

## 先确认的事实

- H5 状态机以当前 API 文档为准。
- 支付金额由服务端计算，前端不传最终应付金额。
- `EXPIRED` 是否允许重发起，以文档约定为准。
- 打到订单上的实收金额，必须能追溯到支付尝试或人工核销记录。

## 实施顺序

1. 确认订单当前可支付状态。
2. 计算应付金额，并明确金额公式。
3. 建立支付尝试或待核销记录。
4. 处理回调或人工确认。
5. 入账并回写订单状态、实收金额和必要日志。

## 强约束

- 支付回调必须幂等。
- 同一业务动作不得重复累计实收金额。
- 订单状态、支付状态、支付流水三者必须能解释同一事实。
- 现金核销和线上回调都必须走统一的金额校验。
- 不允许前端传入可篡改的最终支付金额。

## 本项目重点检查

- H5 `GET /pay/:token`、`POST /pay/:token/initiate`、`GET /pay/:token/status` 是否使用同一状态机。
- `payment_orders` 与 `payments` 是否各司其职。
- 订单 `paidAmount` 的更新是否只来自已确认入账动作。
- 失败、超时、过期、重复回调是否都有明确处理分支。

## 完成标准

- 支付链路可解释、可追溯、可幂等。
- 金额累计不依赖偶然顺序。
- 文档、实现、数据模型中的支付语义完全一致。
