# 租户边界与权限

先回答“谁能看什么”，再写接口。

## 适用场景

- 设计或审查 Admin、Tenant、H5 三端作用域和权限边界。
- 修改需要自动按 `tenantId` 过滤数据的接口和模块。

## 必做顺序

1. 识别接口属于平台侧、租户侧还是 H5。
2. 明确当前 token 中的 `tenantId`、`role`、必要时的 `side`。
3. 检查查询条件、更新条件、唯一键是否都带对了租户边界。
4. 确认平台端不会误用租户侧的默认过滤。

## 硬约束

- 平台用户 `tenantId = null`，租户用户 `tenantId != null`。
- Tenant 侧不得依赖前端传来的 `tenantId` 做最终授权判断。
- service 和 Prisma 查询不得遗漏租户过滤。
- Admin 侧跨租户能力必须显式，不能误复用租户侧默认作用域。
- H5 侧仅通过 `qrCodeToken` 路由订单，不承载后台登录态。
- 不得混用平台角色名、租户角色名和历史旧值。

## 完成标准

- 任一接口都能清楚回答“当前用户能看哪些数据、为什么”。
- 平台侧和租户侧查询不会串边界。
- 角色和租户作用域在文档、JWT claims、实现中一致。
