# 租户边界与权限

先控制数据边界，再讨论功能是否可用。多租户项目里，作用域错误比功能缺失更危险。

## 入口约束

1. 先遵守仓库根目录 `AGENTS.md`。
2. 若当前执行手册未进入某个业务域，不提前扩散该域的权限模型。
3. 平台侧、租户侧、H5 侧边界以当前 API 文档和执行手册为准，不参考历史归档方案。

## 边界原则

- 平台用户 `tenantId = null`，具有跨租户视角。
- 租户用户 `tenantId != null`，默认只看本租户数据。
- 前端不应通过传参覆盖后端判定出的租户作用域。
- 固定角色是治理工具，不是随意扩张的配置项。

## 审查顺序

1. 识别接口属于平台侧、租户侧还是 H5。
2. 明确当前 token 中的 `tenantId`、`role`、必要时的 `side`。
3. 检查查询条件、更新条件、唯一键是否都带对了租户边界。
4. 确认平台端不会误用租户侧的默认过滤。

## 本项目重点

- Tenant 侧列表、详情、更新接口都应自动注入 `tenantId`。
- Admin 侧跨租户查看接口应显式声明数据范围。
- H5 侧公开接口仅通过 `qrCodeToken` 路由订单，不承载后台登录态。
- 固定角色与只读角色接口以当前 API 文档为准，不再走历史动态角色设计。

## 禁止事项

- 依赖前端传来的 `tenantId` 做最终授权判断。
- 在 service 或 Prisma 查询里遗漏租户过滤。
- 混用平台角色名、租户角色名和历史旧值。

## 完成标准

- 任一接口都能清楚回答“当前用户能看哪些数据、为什么”。
- 平台侧和租户侧查询不会串边界。
- 角色和租户作用域在文档、JWT claims、实现中一致。
