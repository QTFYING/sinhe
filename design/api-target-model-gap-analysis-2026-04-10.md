> 日期：2026-04-10
> 最近同步：2026-04-11
> 文档状态：当前生效
> 文档定位：差异桥接手册
> 目标：识别当前仓库实现与现行 API 文档、枚举、contracts、数据模型参考之间的剩余差异，为后续收口提供明确落点

## 一、使用说明

本清单继续只回答 4 个问题：

1. 当前实现到了什么程度
2. 目标模型还要求什么
3. 剩余差异在哪里
4. 后续应当“收口 / 扩展 / 暂缓”哪一类实现

本清单不替代 [api-implementation-delivery-plan-2026-04-10.md](/D:/Sinhe/api/design/api-implementation-delivery-plan-2026-04-10.md)，而是作为其“现状与剩余差异”的桥接文档。

## 二、总体判断

截至当前，仓库的主要断层已经从“核心主链路缺失”转为“平台侧、联调产物和工程化收口不足”。

当前总体判断如下：

1. 租户侧主链路已经基本按现行 API 文档重建完成。
2. 核心 schema 已经能支撑订单、支付、打印配置与打印回执。
3. 当前剩余差异主要集中在：
   - Admin 与外围域尚未系统化落地
   - Swagger / OpenAPI 尚未与已实现接口系统同步
   - 个别能力仍以阶段性简化实现，例如 `reminder`
   - 导入链路具备可用性，但还不具备上线级可靠性

## 四、分领域差异分析

### 4.1 Auth

当前状态：

- 已实现 `login / refresh / logout / me`
- 认证机制与租户上下文可继续复用

剩余差异：

1. Swagger 产物未同步到当前真实结构
2. 仍需在后续联调中复核响应体与枚举显示

结论：

- Auth 已不是主风险点
- 后续以产物同步为主，不再重写

### 4.2 Prisma 核心模型

当前状态：

- 核心 schema 已与当前订单、支付、打印、设置主链路匹配

剩余差异：

1. 需随平台侧与技术债回补继续做增量补模
2. 催款提醒等尚未进入模型

结论：

- Prisma 核心模型已脱离“待重写”阶段
- 后续重点是按新增已确认业务补充，而不是再次整体改写

### 4.3 Settings / General

当前状态：

- “平台默认 + 租户覆盖”模型已落地

剩余差异：

1. Swagger 未同步
2. 需在联调中复核默认值展示与覆盖行为

结论：

- 保持现状，进入联调与产物同步阶段

### 4.4 Settings / Printing

当前状态：

- `GET /settings/printing`
- `GET /settings/printing/:importTemplateId`
- `PUT /settings/printing/:importTemplateId`
  已按 `tenantId + importTemplateId` 模型落地

剩余差异：

1. Swagger 未同步
2. 需在联调中确认列表、详情、默认回退行为

结论：

- 打印配置链路已完成主实现
- 旧 `/print/jobs` 不再属于任何后续补齐范围

### 4.5 Import Template / Import

当前状态：

- 已具备模板与订单导入主链路
- 已能支撑订单导入和打印模板绑定

剩余差异：

1. 导入任务仍是进程内异步，未接入持久化队列
2. 模板自动匹配仍为 best-effort 策略
3. 预检结果仅缓存于 Redis，缺少长期可审计载体

结论：

- Import 已从“未实现”变为“可用但需可靠性增强”
- 相关问题纳入技术债处理，而不是再次推倒重写

### 4.6 Orders

当前状态：

- Orders 已成为主数据源
- 创建、更新、列表、详情、导入、打印回执、收款补充动作已围绕新模型落地

剩余差异：

1. Swagger 未同步
2. `reminder` 当前只有接口回执，尚未持久化

结论：

- Orders 主域已完成主重建
- 订单侧的主要待办已缩小为补强类工作

### 4.7 H5 Payment

当前状态：

- H5 支付链路已围绕 `payment_orders + payments + orders` 重建

剩余差异：

1. Swagger 未同步
2. 仍需结合真实支付渠道联调验证回调与状态轮询口径

结论：

- Payment 已脱离“按旧 `payStatus` 重写”的阶段
- 后续重点在联调验证和平台对账能力，而不是再次改义

### 4.8 Print Records

当前状态：

- `POST /orders/print-records` 已落地
- 打印次数与批次幂等模型已建立

剩余差异：

1. Swagger 未同步
2. 需在联调中复核重复提交回放结果

结论：

- Print Records 已收口
- 后续只做稳定性验证，不再引回打印任务式接口

### 4.9 Notification

当前状态：

- 已完成最小可编译收口

剩余差异：

1. 仍缺真实通知通道、发送留痕和更完整的业务闭环
2. 与 `reminder` 持久化的衔接尚未完成

结论：

- Notification 不再是“冻结不动”的状态
- 但也不应在当前优先级上压过 Swagger、Admin 和主链路技术债

### 4.10 Report

当前状态：

- 已完成最小可编译收口

剩余差异：

1. 需在真实订单与支付数据口径下复核统计结果
2. Swagger 与查询边界未系统整理

结论：

- Report 可继续保留最小实现
- 等平台对账与联调稳定后再做指标校准

### 4.11 Admin 主域

当前状态：

- 已完成第一轮平台侧实现，包含：
  - `/tenants*`
  - `/users*`
  - Admin 视角 `/orders`
  - Admin 视角 `/payments`
  - `/reconciliation*`

剩余差异：

1. Swagger 与最小联调验证尚未同步
2. 少量“动作已落地但未单独建独立记录表”的链路仍需后续增强

结论：

- Admin 主域已从“主线缺口”转为“联调与增强阶段”

### 4.12 Swagger / OpenAPI

当前状态：

- 尚未形成覆盖当前已完成接口的系统化产物

剩余差异：

1. 已完成实现缺少统一联调说明面
2. 新旧注解若不同步，后续会造成二次漂移

结论：

- Swagger 是当前最应立即进入的收口工作，而不是可无限后置的附属项

## 五、旧链路清退对象

以下对象继续明确退出主链路，不再作为未来扩展基础：

1. `/print/jobs`
2. `erpOrderNo`
3. 旧 `templateId`
4. 旧 `customFields`
5. 基于旧 `payStatus` 的支付主流程
6. 服务端组装打印聚合链接的旧思路
7. 任何中文枚举值作为事实值的实现方式

## 六、处理策略总原则

### 6.1 保持稳定

- Auth
- Prisma 核心模型
- Settings / General
- Settings / Printing
- Orders
- H5 Payment
- Print Records

### 6.2 继续扩展

- Admin 主域
- Tenant 外围域
- Notification
- Report

### 6.3 立即收口

- Swagger / OpenAPI
- `reminder` 持久化
- 导入可靠性技术债

### 6.4 暂不优先

- 套餐、合同、工单、Ops 等更外围域
- 上线级优化之外的体验类微调

## 七、执行判定规则

从当前开始，遇到实现优先级分歧时，按以下顺序判断：

1. 已完成主链路若只差产物同步，优先补产物，不再回头重写。
2. 若缺口会影响联调、审计、幂等或可靠性，优先高于新增外围功能。
3. 平台侧接口必须在 Admin 边界下实现，不得复用租户默认作用域“偷跑”。
4. 任何新增接口若会引入旧字段或旧路径，应直接判定为不进入当前主线。

## 八、结论

当前仓库已经不处于“目标模型与实现完全断裂”的阶段。
接下来的重点不是继续证明主链路能做出来，而是：

1. 把已完成主链路同步成可联调产物
2. 把平台侧主域补齐
3. 把少数明确技术债收口
4. 让后续开发重新回到“按文档稳定迭代”的节奏
