# 接口实现分层清单

> 日期：2026-04-11
> 文档状态：当前生效
> 文档定位：接口编码执行手册
> 适用范围：`docs/api`、`packages/types/src/contracts`、`apps/api`
> 目标：把已经稳定的 API 文档，持续转成可维护的后端实现、联调产物和后续平台侧能力

## 一、当前状态

截至当前，项目已经完成租户侧主链路重建，接口实现不再处于“从零开写”的阶段。

已完成：

1. `Auth`
2. Prisma 核心模型
3. `settings/general`
4. `settings/printing*`
5. Import Template / Import
6. Orders
7. H5 Payment
8. Print Records
9. `notification / report` 最小可编译收口

当前主要未完成项：

1. Swagger / OpenAPI 系统化同步
2. `POST /orders/{id}/reminders` 持久化
3. 导入可靠性增强与质量门禁补齐

因此，从现在开始，接口实现应按“收口优先、平台侧优先、可靠性优先”的顺序推进，而不是继续按租户主链路重复重写。

## 二、实现分层

每个接口落地时，统一按以下 6 层执行。

### L1 契约层

职责：

1. 确认路径、方法、角色、请求体、响应体以 `docs/api/*.md` 为准
2. 确认 `packages/types/src/contracts` 已完成投影
3. 确认该接口不回退到已废弃路径、字段或状态

完成标准：

1. 文档正文可独立阅读
2. `contracts` 名称与文档语义一致
3. 不引用废弃路径或历史字段

### L2 接口层

职责：

1. 定义 controller 路由
2. 定义 DTO、参数校验、路径参数和 query 参数
3. 接入 Guard、角色校验、租户上下文注入

完成标准：

1. controller 只处理 HTTP 契约
2. DTO 不承载业务流程判断
3. Tenant 接口不接受前端传入最终 `tenantId`

### L3 应用服务层

职责：

1. 编排业务流程
2. 驱动状态流转
3. 控制幂等、事务和跨表写入顺序

完成标准：

1. service 只承载业务语义，不直接暴露 Prisma 细节
2. 跨表写入链路显式收口
3. 状态迁移与文档一致

### L4 领域规则层

职责：

1. 统一处理状态机、金额计算、权限前置规则
2. 抽离可复用规则，例如：
   - 订单是否允许作废
   - 支付单是否允许重新发起
   - 现金核销是否允许执行
   - 打印回执是否重复

完成标准：

1. 规则可被复用
2. 不把核心状态判断散落在多个 controller / service 分支里

### L5 持久化层

职责：

1. 落 Prisma 查询与写入
2. 明确唯一键、组合唯一、索引的用法
3. 必要时使用事务

完成标准：

1. 查询条件带对租户边界
2. 组合唯一键和幂等键被实际利用
3. 不引用旧 Prisma 字段名

### L6 产物层

职责：

1. Swagger / OpenAPI 同步
2. 最小联调验证
3. 阶段性遗留项记录

完成标准：

1. Swagger 服从文档，不反向改义
2. 至少完成局部类型检查、构建或接口自测
3. 明确写出暂缓项

## 三、当前接口落地顺序

后续接口实现不再按原始主链路顺序推进，而按当前剩余差异推进。

### 第一层：联调产物收口层

目标：

先把已经完成的主链路同步为稳定联调产物。

应处理对象：

1. Auth Swagger
2. Settings / General Swagger
3. Settings / Printing Swagger
4. Import Template / Import Swagger
5. Orders Swagger
6. H5 Payment / Payments Swagger
7. Print Records Swagger

验收：

1. Swagger 路径、请求体、响应体与文档一致
2. 不出现旧路径和旧字段
3. 枚举值与 `packages/types/src/enums` 一致

### 第二层：主链路补强层

目标：

把已完成主链路中的明显缺口收口到可联调、可审计状态。

应实现事项：

1. `POST /orders/{id}/reminders` 持久化
2. 打印回执重复提交回放验证
3. 支付回调与状态查询联调复核

验收：

1. 核心动作可追溯
2. 金额、状态与幂等口径稳定
3. 不为补强动作引回旧模型

### 第三层：可靠性与质量层

目标：

把当前“可编译、可运行”的实现提升到“可联调、可持续迭代”的状态。

应处理对象：

1. 导入任务队列化
2. 导入预检快照策略
3. 关键链路最小自测
4. 阶段性回归记录

验收：

1. 关键任务不依赖单进程偶然成功
2. 核心链路具备最小验证依据
3. 技术债被有计划地消化

## 四、单接口实施检查表

每实现或收口一个接口，统一按以下清单打勾：

1. 路径、方法、角色是否与 `docs/api` 一致
2. `contracts` 是否已存在或已同步更新
3. DTO 是否只做参数校验
4. controller 是否只承接 HTTP 参数
5. service 是否收口业务流程
6. 是否需要事务
7. 是否需要幂等
8. 是否需要 tenant scope
9. 是否涉及状态机
10. 是否涉及金额计算
11. 是否需要同步 Swagger
12. 是否已记录遗留项

## 五、当前推荐执行顺序

从现在开始，建议严格按以下顺序继续推进：

1. Swagger / OpenAPI 第一轮系统化同步
2. `POST /orders/{id}/reminders` 持久化
3. 导入可靠性技术债
4. 关键链路自测与阶段性回归

## 六、非目标

本清单当前不要求立即完成以下事项：

1. 为了补一个外围接口而回退已稳定主链路语义
2. 在平台主域未明确前先大面积铺开更外围模块
3. 恢复任何旧 RPC 路径或 `/print/jobs`
4. 让 Swagger 反向影响文档和 contracts

## 七、收口原则

每完成一层，必须完成对应收口：

1. 文档一致性检查
2. 局部类型检查或构建验证
3. 关键路径最小自测或联调记录
4. 遗留项写入设计文档或技术债清单
