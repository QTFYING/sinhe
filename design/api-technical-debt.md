# API 技术债台账

> 日期：2026-04-11
> 文档状态：当前生效
> 文档定位：实施期技术债清单
> 适用范围：`apps/api`、`design`、`docs/api` 对应的后端落地工作
> 目标：记录当前阶段“为了先打通主链路而暂未收口”的问题，避免后续遗忘、漂移或重复讨论

## 一、使用原则

本清单只记录 3 类事项：

1. 当前实现已经明确存在缺口，但不阻塞主链路继续推进
2. 当前为了降低复杂度做了阶段性简化，后续必须回补
3. 当前已有实现可用，但在可靠性、一致性、可运维性上仍不达标

不纳入本清单的事项：

1. 纯猜测风险
2. 尚未进入实施阶段的功能脑暴
3. 与当前事实源无关的历史遗留讨论

## 二、当前技术债总览

| 编号 | 主题 | 当前状态 | 优先级 | 建议处理阶段 |
|---|---|---|---|---|
| TD-001 | 导入任务尚未接入外部 MQ / 重试死信机制 | 已完成 `import_jobs + 独立 Worker + Redis 锁` 收口 | 中 | 测试环境稳定后、上线前 |
| TD-002 | 导入模板自动匹配规则仍为宽松命中策略 | 当前按列头命中数 best-effort 选择模板 | 中 | 导入链路稳定后 |
| TD-003 | 导入预检结果仅缓存于 Redis | 当前依赖 TTL 快照，不可审计、不可回放 | 中 | 队列化导入时一并评估 |
| TD-004 | 全仓质量门禁尚未系统化恢复 | 当前已具备 `build + smoke`，但更完整的集成回归仍未形成体系 | 中 | 主链路完成后持续收口 |

## 三、逐项说明

### TD-001 导入任务尚未接入外部 MQ / 重试死信机制

**现状：**

- 当前 `POST /orders/import` 会先创建 `import_jobs`
- 正式导入所需快照、冲突策略和进度已持久化到 `import_jobs`
- Web API 进程默认不再消费导入任务
- 独立 `import-worker` 进程会轮询 `pending / stale processing` 任务，并通过 Redis 分布式锁恢复执行
- 当前已经不再依赖 `setImmediate` 裸异步直跑，也不再要求正式导入强依赖 Redis 里的预检快照常驻

**风险：**

1. 当前队列承载仍基于 `import_jobs`，尚未引入外部 MQ
2. 失败重试次数、退避策略和死信处理尚未系统化
3. 超大批量导入时，仍缺少更细的并发治理与观测能力

**建议方案：**

1. 若测试环境规模继续增大，再评估引入真正的任务队列，例如 BullMQ
2. 保持 `POST /orders/import` 只负责创建 `import_jobs`
3. 在现有 Worker 基础上补失败重试次数、退避策略、并发度和最终失败口径
4. 视运维需求再决定是否引入死信队列与任务监控面板

**触发条件：**

- 准备进入测试环境长期联调
- 计划部署到多实例环境
- 需要处理大批量导入任务

**验收标准：**

1. 外部 MQ 是否值得引入有明确结论
2. `import_jobs.status` 不会长期卡死在中间态
3. 重试、退避与最终失败记录可解释

### TD-002 导入模板自动匹配规则仍为宽松命中策略

**现状：**

- 当前未显式传 `templateId` 时，服务端会根据列头命中数选择最佳模板
- 该策略实现简单，但在模板较多、列头相似时可能命中错误模板

**风险：**

1. 多套模板字段接近时，可能自动命中到非预期模板
2. 错误命中的结果会把预检错误前移为“数据异常”，不利于定位

**建议方案：**

1. 增加“必填字段命中率 + 总命中率”的双维度判定
2. 设置最低匹配阈值，低于阈值时直接要求前端显式传 `templateId`
3. 必要时返回候选模板列表，而不是直接替用户决定

**触发条件：**

- 一个租户内导入模板数量明显增加
- 前端反馈模板误识别问题

**验收标准：**

1. 模板误匹配概率显著下降
2. 未达到阈值时不再“猜一个模板继续跑”

### TD-003 导入预检结果仅缓存于 Redis

**现状：**

- 当前 `previewId` 对应的预检快照存放在 Redis
- 正式导入提交后，服务端会把本次导入所需快照同步写入 `import_jobs.snapshot`
- 但“预检阶段本身”的完整结果仍只存在 Redis，适合短时复用，不适合长期审计、复盘和失败重放

**风险：**

1. Redis TTL 到期后，前端提交旧 `previewId` 会直接失效
2. 无法追踪某次导入前实际看到的完整预检结果
3. 若后续需要审计“预检页面给用户看到了什么”，当前仍缺少独立持久化载体

**建议方案：**

1. 若后续只需要短时联调复用，维持现状即可
2. 若需要审计和重放，可新增 `import_previews` 表持久化关键快照
3. 也可采取折中方案：Redis 保留完整快照，数据库仅保存摘要和校验指纹

**触发条件：**

- 队列化导入落地
- 业务要求追踪某次导入前的预检结果
- 需要支持失败重试和回放

**验收标准：**

1. `previewId` 生命周期与正式导入链路一致
2. 关键预检结果可查询、可解释

### TD-004 全仓质量门禁尚未系统化恢复

**现状：**

- 当前 `apps/api` 主链路已按新契约重建
- `pnpm -F api build` 与 `pnpm -F api test:smoke` 已通过
- smoke 已覆盖导入 Worker、导入模板匹配、导入订单推导、订单状态推导、账期状态推导、H5 支付状态推导与支付超时过期判定
- 但更完整的集成回归、联调记录与阶段性回归基线仍未形成系统化质量门禁

**风险：**

1. 已实现接口缺少统一联调产物，容易在联调阶段再出现口径漂移
2. 缺少最小回归依据，后续新增平台侧或外围域时容易引入隐性回归
3. 当前 build 通过并不等于主链路已经具备可验证、可交付的质量基线

**建议方案：**

1. 保持已完成主链路与外围域 Swagger 持续同步
2. 在现有 smoke 基础上，为 Auth、Import、Orders、Payment 增加更贴近业务的回归覆盖
3. 在平台侧主域推进过程中，逐步恢复更完整的质量门禁，而不是只看单次 build 结果

**触发条件：**

- 当前主链路编码已完成，准备进入联调和平台侧扩展

**验收标准：**

1. 已完成域都有与文档一致的 Swagger 产物
2. 关键链路具备最小联调或回归依据
3. 后续新增实现不再只依赖“能 build 过”作为唯一完成标准

## 四、建议处理顺序

建议按以下顺序收口：

1. 队列化导入进一步增强时，一并评估 `TD-003`
2. 模板自动匹配优化 `TD-002` 放在导入体验优化阶段
3. `TD-004` 作为当前所有后续阶段的共同收口项，不再后置
4. `TD-001` 仅在测试环境暴露出更高运维复杂度时再提升优先级

## 五、当前结论

当前最值得立即记住的结论是：

1. 导入功能已经完成独立 Worker 收口，但外部 MQ 与预检持久化仍未最终定稿
2. 当前最主要的技术债已经聚焦为“导入预检持久化、模板匹配优化、质量门禁继续细化”
3. 催款提醒持久化已完成，不再属于未收口主链路缺口
4. 下一步应优先处理更完整的联调回归与导入体验类技术债
