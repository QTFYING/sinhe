# API 技术债清单

> 日期：2026-04-11
> 文档状态：当前生效
> 文档定位：实施期技术债清单
> 适用范围：`apps/api`、`design`、`docs/api` 对应的后端落地工作
> 目标：记录当前阶段“为了先打通主链路而暂未收口”的问题，避免后续遗忘、漂移或重复讨论

## 一、使用原则

本清单只记录 3 类事项：

1. 当前实现已经明确存在缺口，但不阻塞主链路继续推进
2. 当前为了降低复杂度做了阶段性简化，后续必须回补
3. 当前已有实现可用，但在可靠性、一致性、可运维性上仍不达标

不纳入本清单的事项：

1. 纯猜测风险
2. 尚未进入实施阶段的功能脑暴
3. 与当前事实源无关的历史遗留讨论

## 二、当前技术债总览

| 编号 | 主题 | 当前状态 | 优先级 | 建议处理阶段 |
|---|---|---|---|---|
| TD-001 | 导入任务未接入真正持久化队列 | 当前为进程内异步处理 | 高 | 订单主域稳定后、上线前 |
| TD-002 | 导入模板自动匹配规则仍为宽松命中策略 | 当前按列头命中数 best-effort 选择模板 | 中 | 导入链路稳定后 |
| TD-003 | 导入预检结果仅缓存于 Redis | 当前依赖 TTL 快照，不可审计、不可回放 | 中 | 队列化导入时一并评估 |
| TD-004 | 全仓质量门禁尚未系统化恢复 | 当前 `apps/api` 已可 build，但 Swagger、自测与阶段性回归仍未形成体系 | 高 | 主链路完成后立即收口 |

## 三、逐项说明

### TD-001 导入任务未接入真正持久化队列

**现状：**

- 当前 `POST /orders/import` 会先创建 `import_jobs`
- 随后通过进程内异步方式启动导入处理
- 这能满足本地开发和单实例联调，但不具备可靠任务投递能力

**风险：**

1. 服务重启后，任务可能停留在 `pending` 或 `processing`
2. 多实例部署时，无法保证任务调度一致性
3. 任务执行过程不可恢复，不具备重试和死信处理能力

**建议方案：**

1. 引入真正的任务队列，例如 BullMQ
2. `POST /orders/import` 只负责创建 `import_jobs` 并投递 job
3. Worker 进程负责消费任务并更新 `import_jobs`
4. 明确失败重试次数、退避策略和最终失败口径

**触发条件：**

- 准备进入测试环境长期联调
- 计划部署到多实例环境
- 需要处理大批量导入任务

**验收标准：**

1. 服务重启后任务不丢失
2. `import_jobs.status` 不会长期卡死在中间态
3. 重试与失败记录可解释

### TD-002 导入模板自动匹配规则仍为宽松命中策略

**现状：**

- 当前未显式传 `templateId` 时，服务端会根据列头命中数选择最佳模板
- 该策略实现简单，但在模板较多、列头相似时可能命中错误模板

**风险：**

1. 多套模板字段接近时，可能自动命中到非预期模板
2. 错误命中的结果会把预检错误前移为“数据异常”，不利于定位

**建议方案：**

1. 增加“必填字段命中率 + 总命中率”的双维度判定
2. 设置最低匹配阈值，低于阈值时直接要求前端显式传 `templateId`
3. 必要时返回候选模板列表，而不是直接替用户决定

**触发条件：**

- 一个租户内导入模板数量明显增加
- 前端反馈模板误识别问题

**验收标准：**

1. 模板误匹配概率显著下降
2. 未达到阈值时不再“猜一个模板继续跑”

### TD-003 导入预检结果仅缓存于 Redis

**现状：**

- 当前 `previewId` 对应的预检快照存放在 Redis
- 适合短时复用，但不适合长期审计、复盘和失败重放

**风险：**

1. Redis TTL 到期后，前端提交旧 `previewId` 会直接失效
2. 无法追踪某次导入前实际看到的预检结果
3. 队列化后若导入处理被延迟，预检上下文与缓存生命周期可能脱钩

**建议方案：**

1. 若后续只需要短时联调复用，维持现状即可
2. 若需要审计和重放，可新增 `import_previews` 表持久化关键快照
3. 也可采取折中方案：Redis 保留完整快照，数据库仅保存摘要和校验指纹

**触发条件：**

- 队列化导入落地
- 业务要求追踪某次导入前的预检结果
- 需要支持失败重试和回放

**验收标准：**

1. `previewId` 生命周期与正式导入链路一致
2. 关键预检结果可查询、可解释

### TD-004 全仓质量门禁尚未系统化恢复

**现状：**

- 当前 `apps/api` 主链路已按新契约重建
- `pnpm -F api build` 已通过
- 但 Swagger / OpenAPI、关键链路自测、阶段性回归记录仍未形成系统化质量门禁

**风险：**

1. 已实现接口缺少统一联调产物，容易在联调阶段再出现口径漂移
2. 缺少最小回归依据，后续新增平台侧或外围域时容易引入隐性回归
3. 当前 build 通过并不等于主链路已经具备可验证、可交付的质量基线

**建议方案：**

1. 先为已完成主链路补齐 Swagger / OpenAPI
2. 为 Auth、Settings、Orders、Payment、Print Records 建立最小自测或联调记录
3. 在平台侧主域推进过程中，逐步恢复更完整的质量门禁，而不是只看单次 build 结果

**触发条件：**

- 当前主链路编码已完成，准备进入联调和平台侧扩展

**验收标准：**

1. 已完成域都有与文档一致的 Swagger 产物
2. 关键链路具备最小联调或回归依据
3. 后续新增实现不再只依赖“能 build 过”作为唯一完成标准

## 四、建议处理顺序

建议按以下顺序收口：

1. 在准备进入更稳定联调或测试环境前，优先处理 `TD-001`
2. 队列化导入设计时，一并评估 `TD-003`
3. 模板自动匹配优化 `TD-002` 放在导入体验优化阶段
4. `TD-004` 作为当前所有后续阶段的共同收口项，不再后置

## 五、当前结论

当前最值得立即记住的结论是：

1. 导入功能已经具备契约可用性，但还不具备上线级可靠性
2. 当前最主要的技术债已经聚焦为“导入可靠性”和“质量门禁继续细化”
3. 催款提醒持久化已完成，不再属于未收口主链路缺口
4. 下一步应优先处理导入可靠性和更完整的联调回归
