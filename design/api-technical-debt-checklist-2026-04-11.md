# API 技术债清单

> 日期：2026-04-11
> 文档状态：当前生效
> 文档定位：实施期技术债清单
> 适用范围：`apps/api`、`design`、`docs/api` 对应的后端落地工作
> 目标：记录当前阶段“为了先打通主链路而暂未收口”的问题，避免后续遗忘、漂移或重复讨论

## 一、使用原则

本清单只记录 3 类事项：

1. 当前实现已经明确存在缺口，但不阻塞主链路继续推进
2. 当前为了降低复杂度做了阶段性简化，后续必须回补
3. 当前已有实现可用，但在可靠性、一致性、可运维性上仍不达标

不纳入本清单的事项：

1. 纯猜测风险
2. 尚未进入实施阶段的功能脑暴
3. 与当前事实源无关的历史遗留讨论

## 二、当前技术债总览

| 编号 | 主题 | 当前状态 | 优先级 | 建议处理阶段 |
|---|---|---|---|---|
| TD-001 | 导入任务未接入真正持久化队列 | 当前为进程内异步处理 | 高 | 订单主域稳定后、上线前 |
| TD-002 | 导入模板自动匹配规则仍为宽松命中策略 | 当前按列头命中数 best-effort 选择模板 | 中 | 导入链路稳定后 |
| TD-003 | 导入预检结果仅缓存于 Redis | 当前依赖 TTL 快照，不可审计、不可回放 | 中 | 队列化导入时一并评估 |
| TD-004 | 旧 MVP 模块仍造成全量类型检查失败 | 当前 `order/payment/print/report/notification` 仍未按新契约重建 | 高 | 按主线阶段逐批清退 |
| TD-005 | 催款提醒仅完成接口回执，未做持久化 | 当前只校验订单归属并返回发送结果 | 中 | 支付主域稳定后、通知域联调前 |

## 三、逐项说明

### TD-001 导入任务未接入真正持久化队列

**现状：**

- 当前 `POST /orders/import` 会先创建 `import_jobs`
- 随后通过进程内异步方式启动导入处理
- 这能满足本地开发和单实例联调，但不具备可靠任务投递能力

**风险：**

1. 服务重启后，任务可能停留在 `pending` 或 `processing`
2. 多实例部署时，无法保证任务调度一致性
3. 任务执行过程不可恢复，不具备重试和死信处理能力

**建议方案：**

1. 引入真正的任务队列，例如 BullMQ
2. `POST /orders/import` 只负责创建 `import_jobs` 并投递 job
3. Worker 进程负责消费任务并更新 `import_jobs`
4. 明确失败重试次数、退避策略和最终失败口径

**触发条件：**

- 准备进入测试环境长期联调
- 计划部署到多实例环境
- 需要处理大批量导入任务

**验收标准：**

1. 服务重启后任务不丢失
2. `import_jobs.status` 不会长期卡死在中间态
3. 重试与失败记录可解释

### TD-002 导入模板自动匹配规则仍为宽松命中策略

**现状：**

- 当前未显式传 `templateId` 时，服务端会根据列头命中数选择最佳模板
- 该策略实现简单，但在模板较多、列头相似时可能命中错误模板

**风险：**

1. 多套模板字段接近时，可能自动命中到非预期模板
2. 错误命中的结果会把预检错误前移为“数据异常”，不利于定位

**建议方案：**

1. 增加“必填字段命中率 + 总命中率”的双维度判定
2. 设置最低匹配阈值，低于阈值时直接要求前端显式传 `templateId`
3. 必要时返回候选模板列表，而不是直接替用户决定

**触发条件：**

- 一个租户内导入模板数量明显增加
- 前端反馈模板误识别问题

**验收标准：**

1. 模板误匹配概率显著下降
2. 未达到阈值时不再“猜一个模板继续跑”

### TD-003 导入预检结果仅缓存于 Redis

**现状：**

- 当前 `previewId` 对应的预检快照存放在 Redis
- 适合短时复用，但不适合长期审计、复盘和失败重放

**风险：**

1. Redis TTL 到期后，前端提交旧 `previewId` 会直接失效
2. 无法追踪某次导入前实际看到的预检结果
3. 队列化后若导入处理被延迟，预检上下文与缓存生命周期可能脱钩

**建议方案：**

1. 若后续只需要短时联调复用，维持现状即可
2. 若需要审计和重放，可新增 `import_previews` 表持久化关键快照
3. 也可采取折中方案：Redis 保留完整快照，数据库仅保存摘要和校验指纹

**触发条件：**

- 队列化导入落地
- 业务要求追踪某次导入前的预检结果
- 需要支持失败重试和回放

**验收标准：**

1. `previewId` 生命周期与正式导入链路一致
2. 关键预检结果可查询、可解释

### TD-004 旧 MVP 模块仍造成全量类型检查失败

**现状：**

- 当前导入模块已按新契约重建
- 但 `order/payment/print/report/notification` 等模块仍保留旧字段和旧状态机
- 因此全量 `tsc` 仍会失败

**风险：**

1. 后续真实问题容易被旧模块噪音掩盖
2. 新模块虽然局部正确，但缺少全仓级质量门禁

**建议方案：**

1. 按执行手册阶段逐批重建旧模块
2. 每完成一个主域，就把该域的旧编译错误清零
3. 在进入联调收口前恢复全量 `build/tsc` 作为硬门禁

**触发条件：**

- 进入订单主域、支付主域、打印主域重建

**验收标准：**

1. 相关旧模块不再引用废弃字段
2. 每完成一个阶段，就减少一批全量编译噪音

### TD-005 催款提醒仅完成接口回执，未做持久化

**现状：**

- 当前 `POST /orders/{id}/reminders` 已按订单归属和渠道参数完成接口收口
- 但服务端尚未建立 reminder 持久化模型
- 当前实现只返回 `sent/channels`，无法沉淀提醒记录

**风险：**

1. 无法追踪某张订单是否已催款、催过几次、最近一次何时发起
2. 无法支撑后续“催款记录列表”“催款频次限制”“催款审计”类需求
3. 若后续接入真实短信或企业微信通道，缺少发送前后状态与失败原因留痕

**建议方案：**

1. 增加订单催款记录表，例如 `order_reminders`
2. 最小字段建议包含：
   - `id`
   - `tenantId`
   - `orderId`
   - `channels`
   - `status`
   - `remark`
   - `operatorId`
   - `sentAt`
   - `createdAt`
3. 若后续接入真实发送通道，再补充：
   - `provider`
   - `providerMessageId`
   - `failureReason`
   - `payload`
4. `POST /orders/{id}/reminders` 应调整为：
   - 先落 reminder 记录
   - 再执行业务发送
   - 最后回写发送结果

**触发条件：**

- 准备接入真实短信 / 企业微信发送
- 前端开始需要展示催款历史
- 业务开始关心催款留痕和频次控制

**验收标准：**

1. 每次催款动作都可追溯到具体订单和操作人
2. 能区分“已发起”“已发送”“发送失败”等状态
3. 后续若增加催款历史查询接口，无需再返工主链路

## 四、建议处理顺序

建议按以下顺序收口：

1. 先继续推进订单主域和支付主域，不被模板匹配优化分散精力
2. 在导入链路即将进入稳定联调阶段时，优先处理 `TD-001`
3. 队列化导入设计时，一并评估 `TD-003`
4. 模板自动匹配优化 `TD-002` 放在导入体验优化阶段
5. `TD-004` 按既定实施阶段持续清理，不单独开分支“救火”
6. `TD-005` 在通知通道落地前完成建模，避免真实发送后仍无记录

## 五、当前结论

当前最值得立即记住的结论是：

1. 导入功能已经具备契约可用性，但还不具备上线级可靠性
2. 当前最主要的技术债不是“功能缺失”，而是“任务可靠性”和“旧模块噪音”
3. 催款提醒当前只完成了接口占位，若要进入真实通知链路必须先补持久化模型
4. 下一步仍应以订单主域和支付主域为主线推进，不要被局部优化打断
