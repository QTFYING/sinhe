经销商订单收款平台 - 业务规划与需求讨论稿
文档版本： V1.0 (内部业务对齐版)
会议目标： 与创始团队及业务骨干对齐产品 V1 版本的核心业务流程、商业闭环，并对关键业务规则进行决策拍板。
一、 产品定位与核心护城河
1. 解决什么痛点？
在快消、酒水、生鲜等大 B 配送行业，长期存在以下顽疾：
资金风险： 司机带静态收款码去送货，钱到底付没付、付给谁了，公司无法实时掌控，存在司机挪款跑路风险。业务员代收货款，造成收款脱节。
财务人效极低： 每天多笔订单，财务需要拿着银行流水和 ERP 里的送货单逐笔“打钩”对账，极易出错。
业务连贯性差： 打单员开单、司机送货、财务收款，三个环节脱节。
2. 我们的核心护城河（V1 卖点）
“一单一码”强绑定： 打印送货单时，自动生成该订单专属的收款二维码。客户扫码只能付这笔订单的钱，金额不可改，且钱直接进代理商绑定的银行账户。
“秒级”自动核销对账： 只要客户付完款，系统立刻标记该订单“已支付”，财务直接看汇总报表，彻底告别人工对账。现金收入清晰知道未到账信息。账期客户自动分类提醒回款。
无缝的“静默打印”体验： 网页上点一下打印，本地打印机直接出单据，跟他们原来用 ERP 打印一样顺滑，不增加打单员的学习成本。
二、 V1 版本核心业务场景走查 (User Journey)
为了让大家更有画面感，我们推演一下标准商户（比如某酒水批发商）的日常使用场景：
接单与导入：
打单员从他们现有的管家婆/金蝶 ERP 中导出今天的送货单（Excel）。
把 Excel 拖入我们的系统，系统自动识别出“订单号、客户名、商品明细、总金额”。
打单出库：
打单员全选今天的 50 笔订单，点击“批量打印”。（增加功能：第几次打印）
旁边的针式/热敏打印机直接呼呼出纸，每张单子底部都印着一个专属的收款二维码。
司机拿走单子，开车去送货。
送货与收款：
司机把 10 箱啤酒卸到小卖部，老板掏出微信/支付宝扫描单子底部的二维码。
手机弹出页面显示：“订单号 xxx，金额 1500 元，收款方：某某酒水批发”。老板指纹支付成功。
实时反馈：
司机刷新一下自己的手机端（或收到短信），看到这单已经“已支付”，放心离开。（是否可以实现？）
公司财务的电脑后台，这笔订单瞬间变成绿色（已收款），对账单自动 +1500 元。
三、 关键业务模块（V1 范围确认）
通道聚合 (支付底层)：
V1 明确接入： 平安银行、拉卡拉、收钱吧。
业务逻辑： 我们不碰钱。客户扫码付的钱，由这三家机构直接清算到批发商老板的银行卡里。我们只负责“发号施令”和“接收成功通知”。
开放平台能力 (粘性利器)：
针对有IT能力的大型经销商，不需要他们每天导 Excel。
提供 API 接口，让他们的 ERP 直接把订单推给我们，我们把支付状态实时推回给他们的 ERP，实现真正的无感融合。
服务商裂变系统 (推广引擎)：
支持地推人员/代理商入驻。
代理商可以看到自己名下推广了多少家商户，今天产生了多少流水，以及自己能拿到多少分润。
四、 核心业务规则探讨（本次会议需决策 ⚠️）
以下问题没有绝对的对错，但会直接影响 V1 版本的开发逻辑和对外销售话术，需要大家在会上拍板定调：
议题 1：收款的“异常场景”容忍度
场景： 客户嫌 1500 元太贵，当场退了 2 箱，只愿意付 1200 元。但单子上的二维码写死了 1500 元。
选项 A（死磕管控）： 坚决不允许改价。司机必须给公司打电话，让打单员在后台修改金额，司机手机上生成新的 1200 元二维码让客户扫。（优点：绝对安全无漏洞；缺点：送货效率受影响，司机可能抱怨）
选项 B（适当放权）： 允许系统设置“部分付款”功能，或者给司机开通一定额度的改价权限。
👉 需要业务端确认：我们 V1 是选 A 还是选 B？   两项功能需要兼顾，代理商对商家的容忍度不同。
议题 2：二维码有效期的设定  有效期90天。收款后显示已支付
场景： 二维码被司机拍个照发给客户，客户说“晚上再付”。
问题： 这个码多久过期？如果 3 天后客户才扫码付款，财务那边算哪天的账？记账至开单日期，跟随单据时间
👉 建议： V1 版本支持租户自定义有效期（如 12小时、24小时），过期后扫码提示“已失效，请联系供应商重新获取”。大家是否同意？单据全部丢失情况下，系统可以补单单据
议题 3：通道费率与我们的盈利模式
微信/支付宝底层会收 0.2% ~ 0.38% 不等的通道费，平安银行/拉卡拉也会有成本。
问题： 我们平台怎么赚钱？
模式一：赚流水差价（跟拉卡拉谈 0.2%，卖给商户 0.25%，赚万五）。
模式二：纯 SaaS 模式，不赚流水差价，只收商户平台年费（如 3999元/年）。
👉 需要探讨： 这决定了财务模块里，要不要给商户展示明细的手续费扣除情况。
模式一二都收
议题 4：服务商（代理商）的结算方式
服务商帮我们拉来了流水，我们要给他们发佣金。
问题： 这笔钱是系统每月自动出报表，我们财务线下打款？还是做到系统里，让他们可以在线提现？
👉 建议： 考虑税务合规和开发成本，V1 版本仅提供“佣金对账单看板”，实际打款由我们财务线下（或走灵活用工平台）转账，后台点击“标记为已打款”即可。是否同意？
同意线下付款，服务商开票给公司。
议题 5：数据导入的容错底线
客户导出的 Excel 经常乱七八糟（有合并单元格、有表头、有脚注）。
👉 决策： V1 版本我们是要求客户严格遵守我们的一种标准模板？还是开发“自适应拖拽映射”功能（开发成本高，但客户体验极好）？
自适应拖拽映射
