## 一、项目背景与立项依据

### 1.1 行业痛点

在快消品、酒水、生鲜等大 B 配送行业中，经销商（批发商）普遍依赖传统纸质对账与人工收款模式，长期存在以下核心问题：

| 痛点类别 | 具体表现 | 业务影响 |
|---|---|---|
| **资金风险** | 货款赊账拖欠、口头承诺无凭证 | 坏账率高，资金回笼周期长 |
| **对账低效** | 人工 Excel 核账，月结对账耗时 | 财务人力成本居高，错漏率高 |
| **重复收款** | 无状态管理，买家可能被重复收款 | 客诉风险，损伤终端客情 |
| **流程断层** | 打单、发车、收款三段割裂 | 业务连贯性差，异常无追溯 |

### 1.2 立项依据

基于上述痛点，本项目立项目标是建设一套面向经销商群体的 **SaaS 化订单收款与对账平台**，以数字化手段串联"下单—出库—配送—收款—对账"的完整业务闭环，将原有依赖人力的低效流程升级为系统驱动的智能化作业模式。

### 1.3 业务核心闭环

```
Excel智能导入  →  批量静默打印(一单一码)  →  终端扫码支付(金额管控)
      ↑                                              ↓
 对账核销报表  ←  代理商分润清算  ←  实时资金直达收款账户
```

## 二、系统整体定位与目标

### 2.1 产品定位

本平台定位为 **多租户 SaaS 架构**，面向三类客群提供差异化服务：

- **平台层（OS端）**: 贵司内部运营团队，掌握全局数据视野与商户管理权限。
- **商户层（Tenant端）**: 实体经销商，作为核心付费用户，使用订单管理、打单收款、财务报表等核心功能。
- **终端层（Consumer端）**: 终端超市老板，仅需扫码付款，无需注册，体验极简。

### 2.2 核心设计原则

- **资金安全**: 平台不触碰资金，所有支付通过持牌第三方支付机构（拉卡拉）直达商户子账户，合规第一。
- **操作防呆**: 关键链路（进件、导入、收款）均设置强校验与状态拦截，降低人为操作失误概率。
- **体验极简**: 终端买家扫码即到支付页，无注册、无跳转、无复杂操作，降低收款摩擦。
- **数据可追溯**: 订单全生命周期操作留存流水日志，满足财务审计需求。

## 三、系统架构设计

### 3.1 整体分层架构

本系统采用经典的**前后端分离 + 多前端应用**架构，后端提供统一的 RESTful API，各前端应用按业务边界物理隔离。

```
┌─────────────────────────────────────────────────────┐
│                    接入层 (Nginx)                    │
│         HTTPS 终结 / 反向代理 / 静态资源托管         │
└──────┬───────────┬──────────────┬────────────────────┘
       │           │              │
┌──────▼──┐  ┌─────▼──────┐  ┌───▼──────────────────┐
│OS运营平台│  │租户管理系统 │  │买家扫码收款H5          │
│(admin)  │  │(tenant)    │  │(pay-h5)              │
└──────┬──┘  └─────┬──────┘  └───┬──────────────────┘
       │           │              │
       └───────────┴──────────────┘
                   │ REST API / WebSocket
       ┌───────────▼──────────────────────────┐
       │          后端服务层 (api)              │
       │   NestJS  |  BullMQ队列  |  Redis缓存 │
       └───────────┬──────────────────────────┘
                   │
       ┌───────────▼──────────────────────────┐
       │        数据持久层                      │
       │   PostgreSQL 主库  |  Redis 缓存/锁   │
       └──────────────────────────────────────┘
                   │
       ┌───────────▼──────────────────────────┐
       │       第三方服务集成                   │
       │  拉卡拉支付  |  短信服务  |  OSS存储   │
       └──────────────────────────────────────┘

       ┌──────────────────────────────────────┐
       │    Electron 静默打印客户端 (本地)       │
       │  WebSocket 接收打印指令 → 调起打印机   │
       └──────────────────────────────────────┘
```

### 3.2 Monorepo 分包设计

基于业务边界的严格划分，整个项目采用 Monorepo 单体仓库模式，拆分为 **4个应用包 + 3个共享包**：

```
project-root/
├── apps/
│   ├── api/              # 后端服务（NestJS）
│   ├── admin/            # OS运营平台（React）
│   ├── tenant/           # 租户管理系统（React）
│   └── pay-h5/           # 买家扫码收款页（Preact/原生）
├── packages/
│   ├── shared-types/     # 前后端共享TS类型与DTO
│   ├── shared-utils/     # 公共工具函数（金额、日期、Token等）
│   └── shared-ui/        # 公共UI组件（admin与tenant复用）
├── pnpm-workspace.yaml
└── turbo.json
```

**各包职责说明：**

| 包名 | 类型 | 使用者 | 核心职责 |
|---|---|---|---|
| `apps/api` | 后端 | 全部前端 | 业务逻辑、数据持久化、支付集成 |
| `apps/admin` | PC前端 | 平台运营人员 | 全局监控、商户审核、代理商管理 |
| `apps/tenant` | PC前端 | 经销商老板及其内部员工 | 订单管理、打单、报表、RBAC权限 |
| `apps/pay-h5` | H5前端 | 终端买家 | 扫码展示订单、唤起支付 |
| `packages/shared-types` | 共享包 | 全部应用 | Prisma类型、DTO接口、枚举值 |
| `packages/shared-utils` | 共享包 | 全部应用 | 纯函数工具、校验逻辑 |
| `packages/shared-ui` | 共享包 | admin + tenant | 复用UI组件库 |

## 五、后端技术选型

### 5.1 核心框架：NestJS + TypeScript

**选型理由：**

NestJS 是基于 Node.js 的企业级后端框架，提供开箱即用的**依赖注入容器、模块化体系和装饰器驱动开发**模式。对于本项目这种涉及多角色、多业务域、多支付通道的复杂 SaaS 系统，NestJS 的架构强约束能有效防止代码在高速迭代过程中腐化为难以维护的"面条代码"，显著降低单兵作战的长期维护成本。

**核心模块规划：**

| 模块名 | 职责说明 |
|---|---|
| `AuthModule` | JWT鉴权、角色守卫、预设角色枚举权限校验 |
| `TenantModule` | 租户生命周期管理、SaaS订阅状态 |
| `OrderModule` | 订单创建、状态机流转、生命周期日志 |
| `ImportModule` | Excel解析、字段映射规则存储（标准字段 + 自定义扩展字段定义）、异步入库队列 |
| `PaymentModule` | 拉卡拉支付集成、Webhook回调处理、主动轮询 |
| `PrintModule` | 打印指令下发（WebSocket推送至Electron客户端） |
| `ReportModule` | 实收报表、流水清算、代理商佣金台账 |
| `NotificationModule` | 站内信读写接口、账期提醒 BullMQ 定时任务调度 |
| `AdminModule` | 平台运营超管功能、穿透性数据查询 |

### 5.2 数据库：PostgreSQL + Prisma

**PostgreSQL 选型理由：**

系统强涉及资金流转与订单状态变更，对数据库的**事务一致性、并发安全**要求极高。PostgreSQL 在 ACID 事务、行级锁、复杂查询优化等方面均优于 MySQL，是金融类业务场景的业界首选。

**Prisma 选型理由：**

Prisma 提供 Schema-first 的开发模式，自动生成完全类型安全的查询客户端。结合 Monorepo 架构中的 `shared-types` 包，前后端可共享同一套数据模型类型，从根本上消除接口联调时的字段不一致问题。

**核心数据域规划：**

- **基础设施域**: `tenant_info`（租户主体）、`agent_info`（服务商）、`user_account`（账号权限）、`import_template_config`（导入模板）

**`user_account` 账号体系设计说明：**

OS用户与Tenant用户共用同一张表，以 `tenantId` 是否为空作为区分依据：

| 账号类型 | `tenantId` | 角色枚举 | 创建方式 |
|---|---|---|---|
| OS平台账号 | `null` | `OS_SUPER_ADMIN` / `OS_OPERATOR` | 仅由超管手动创建，无自助流程 |
| 租户账号 | 有值（所属租户ID） | `TENANT_OWNER` / `TENANT_OPERATOR` / `TENANT_FINANCE` / `TENANT_VIEWER` | 由 `TENANT_OWNER` 在系统内自主添加员工账号 |

**角色权限设计原则：**
- OS侧角色固定为2个，硬编码，无需自定义
- Tenant侧角色固定为4个预设枚举，覆盖经销商常见岗位（老板、打单员、出纳、只读），**不做动态角色自建**，避免过度设计
- 行级安全策略（Section 9.3）中，`tenantId = null` 的OS账号天然具备全局数据视野，无需额外处理
- **订单流转域**: `delivery_order`（主订单）、`delivery_order_item`（商品明细）、`order_lifecycle_log`（操作流水日志）
- **财务对账域**: `payment_transaction`（支付流水，一单多笔）、`agent_commission_ledger`（佣金台账）

### 5.3 缓存与异步队列：Redis + BullMQ

**Redis 使用场景：**

- JWT Token 缓存与黑名单管理
- 热点订单数据缓存，降低数据库查询压力
- **分布式锁**：防止并发场景下同一订单被重复发起支付

**BullMQ 使用场景：**

- Excel 大文件异步解析与批量入库（防止同步处理压挂数据库）
- 支付状态主动轮询任务调度
- 代理商佣金日结定时任务
- **账期到期提醒定时任务**：每日凌晨扫描未付清订单，按阶段（到期前 7/3/1 天 + 到期当天）向 TENANT_OWNER 和 TENANT_FINANCE 推送站内信，写入前校验幂等锁（`biz_order_credit_reminder` 唯一约束），重复执行无副作用

### 5.4 支付集成策略

本平台严格遵循**资金不落平台**的合规原则，所有支付资金通过拉卡拉支付通道直达商户进件子账户，平台仅接收支付结果通知。

**支付可靠性双重保障机制：**

- **被动接收**: 拉卡拉支付成功后通过 Webhook 回调通知后端更新订单状态。
- **主动轮询**: 不单纯依赖 Webhook（存在丢包、延迟风险）。当 pay-h5 页面恢复焦点时（如买家从支付 App 切回），主动向后端发起订单状态查询，后端同步向拉卡拉查询支付结果，确保零掉单。

## 六、前端技术选型

### 6.1 OS运营平台（`apps/admin`）

- **技术栈**: React 18 + Vite + TypeScript + Ant Design 5.x
- **状态管理**: Zustand（轻量，适合单开发者维护）
- **数据请求**: TanStack Query（缓存、自动刷新、加载状态管理）

**核心功能页面：**

| 功能模块 | 说明 |
|---|---|
| 数据总览大盘 | 平台总流水走势、租户活跃度、抽润统计 |
| 商户管理 | 租户开户审核、封号、拉卡拉进件参数绑定 |
| 代理商管理 | 服务商档案、佣金配置与台账查询 |
| 超管操作 | 强制重置密码、越权查看任意租户数据、创建/封禁OS账号 |

### 6.2 租户管理系统（`apps/tenant`）

- **技术栈**: React 18 + Vite + TypeScript + Ant Design 5.x
- **状态管理**: Zustand
- **使用场景**: 纯 PC 端后台，经销商老板及内部员工使用。**司机完全处于系统边界之外**，其唯一职责是送货，并将系统打印的带付款码发货单交给终端超市老板，与系统无任何交互。

**核心功能页面：**

| 功能模块 | 说明 |
|---|---|
| Excel智能导入 | 拖拽列映射设计器、模板记忆、异步批量入库 |
| 订单管理 | 订单列表、状态筛选、手工标记已支付 |
| 打印中心 | 批量选单触发打印（V1浏览器打印，V2对接Electron） |
| 财务报表 | 回款总额、实收率、日/周/月维度核销看板 |
| 账期设置 | 租户自主配置最大账期天数与提醒提前天数 |
| 站内信中心 | 账期到期提醒消息列表，支持已读/未读状态，可点击跳转至对应订单 |
| 员工管理 | 老板账号自主添加/停用员工，按预设角色（打单员/出纳/只读）分配权限 |

**权限系统设计（Phase 1 从简原则）：**

权限控制分菜单级与按钮级双层，但**所有权限范围在代码中硬编码预设**，不提供界面化配置入口。

| 预设角色 | 可访问模块 | 操作权限 |
|---|---|---|
| `TENANT_OWNER`（老板） | 全部模块 | 全部操作 + 员工管理 |
| `TENANT_OPERATOR`（打单员） | 导入、订单管理、打印中心 | 增删改查 |
| `TENANT_FINANCE`（出纳） | 财务报表 | 只读 |
| `TENANT_VIEWER`（只读） | 订单列表、财务报表 | 只读 |

- 角色分配交互：员工管理页内单个下拉框选角色，**不弹出菜单Tree选择器**
- 前端实现：按角色枚举做条件渲染，隐藏无权限的菜单项与操作按钮
- Phase 2如有自定义角色诉求，可在此基础上扩展，不影响现有结构

### 6.3 买家扫码收款页（`apps/pay-h5`）

- **技术栈**: Preact（或纯原生 HTML/CSS/JS）
- **构建工具**: Vite
- **设计原则**: 首屏渲染 < 1秒，零框架负担，无复杂路由

**该页面是用户频次最高的 C 端入口，性能要求最严苛，禁止引入任何重量级依赖。**

**核心页面状态：**

| 状态 | 展示内容 |
|---|---|
| 已付款 | 霸屏大绿标"已付款"，阻断重复支付 |
| 未付款 | 订单金额、商品摘要、支付按钮 |
| 已过期 | 二维码超过有效期阻断页，引导联系经销商 |
| 支付中 | Loading 状态，防止用户重复点击 |

### 6.4 打印中心（`apps/tenant` 内嵌）

打印功能直接集成在 `apps/tenant` 中，无需独立客户端。

- **技术方案**: CSS `@media print` + 隐藏 iframe，适配针式打印机纸张尺寸
- **交互流程**: 批量选单 → 后端返回含二维码的打印数据 → 前端触发浏览器打印对话框 → 用户确认打印

**该方案无需本地安装任何客户端，部署零依赖，满足经销商日常打单需求。**

## 七、工程化与代码库管理策略

### 7.1 Monorepo 方案选型

采用 **pnpm Workspace + Turborepo** 作为 Monorepo 管理方案。

| 工具 | 职责 |
|---|---|
| `pnpm Workspace` | 统一依赖管理，避免多包重复安装 node_modules |
| `Turborepo` | 任务编排与增量缓存构建，只构建有变更的子包 |

### 7.2 代码质量保障

| 工具 | 用途 |
|---|---|
| ESLint + Prettier | 统一代码风格，强制格式化 |
| TypeScript 严格模式 | 全量类型约束，杜绝运行时类型错误 |
| Husky + lint-staged | Git 提交前自动执行代码检查，拦截不合规代码入库 |
| Commitlint | 规范 Git commit message 格式（feat/fix/chore等） |

### 7.3 类型共享策略（核心设计）

`packages/shared-types` 包作为整个工程的**类型中枢**，统一导出：
- Prisma Schema 生成的数据模型类型
- 前后端 API 接口的请求/响应 DTO 定义
- 全局枚举值（订单状态、支付状态、角色类型等）

**这一设计保证：后端改动字段时，所有前端应用在构建阶段即可发现类型错误，将接口联调成本降低至接近零。**

## 八、基础设施与部署方案

### 8.1 服务器规划

| 阶段 | 方案 | 说明 |
|---|---|---|

## 九、数据安全与支付合规设计

### 9.1 身份鉴权

- 后台系统（admin/tenant）采用 **JWT + Refresh Token** 双令牌机制，Access Token 短效（2小时），Refresh Token 长效（7天）存于 HttpOnly Cookie。
- C 端收款页（pay-h5）不设传统登录，以**订单唯一 Hash Token**（URL参数）作为访问凭证，一单一 Token，不可猜测、不可伪造。
- 所有 Token 的黑名单状态通过 Redis 维护，支持服务端主动吊销。

### 9.2 支付安全

- **防重复收款**: 数据库层订单状态字段设置乐观锁，Redis 分布式锁防并发，前端页面状态拦截三重保障。
- **二维码有效期**: 支付码有效期由后端读取订单 `CreatedAt` 计算，超过商户配置阈值（默认90天）强制阻断，前端无法绕过。
- **金额防篡改**: 支付金额由后端在重定向至拉卡拉收银页时服务端定格，前端无权传入金额参数，杜绝客户端篡改金额风险。

### 9.3 数据隔离

- 多租户数据隔离采用**行级租户ID过滤**策略（Row-Level Security），所有数据库查询在 ORM 层统一注入 `tenantId` 条件，禁止跨租户数据访问。
- 敏感配置（拉卡拉密钥、数据库连接串）通过环境变量注入，禁止硬编码进代码仓库。

## 十、核心技术难点与攻坚策略

### 10.1 Excel 智能自适应导入

**难点**: 经销商历史数据格式千差万别，含合并单元格、无效表头表尾，无法用固定模板覆盖。

**攻坚策略**:
1. 使用 SheetJS 在浏览器端完成 Excel 解析，不上传原文件，降低服务端压力。
2. 构建可视化拖拽列映射设计器，让用户自主完成字段关联（首次）。
3. 将映射规则序列化持久化至数据库，同格式表格后续导入实现100%自动适配。
4. **自定义字段扩展**：模板中 `customFieldDefs` 声明若干自定义列（如备注1、备注2），浏览器端解析时在完成标准字段映射后，额外提取这些列并组装为 `customFields` JSON 对象，随订单数据一同提交入库。订单数据库以 `Json` 列存储，前端在按模板筛选时动态渲染对应列头，实现无 Schema 变更的字段扩展。
5. 前端校验完成后分 Chunk 批量提交，后端通过 BullMQ 队列异步入库，防止大数据量冲垮数据库。

### 10.2 一单一码打印方案

**方案**: CSS `@media print` + 隐藏 iframe，适配针式打印机纸张尺寸。批量选单后触发浏览器打印对话框，用户点击一次确认即可完成打印，无需安装任何本地客户端。

### 10.3 支付状态零掉单保障

**难点**: 单纯依赖 Webhook 回调存在丢包、延迟等不可控风险，可能导致实际已付款但订单状态未更新。

**攻坚策略**:
- Webhook 被动接收 + 主动轮询双链路并行，任意一路成功即完成状态更新。
- 关键状态变更操作写入 `order_lifecycle_log`，具备完整审计追溯能力。
